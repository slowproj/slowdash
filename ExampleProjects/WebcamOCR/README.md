# WebcamOCR — LLM-based monitoring for screen-only instruments

End-to-end demo for [slowdash-agent-tools](../../slowdash-agent-tools).
A Claude vision model reads the front display of an **Omega CN1500
multi-zone ramp & soak controller** from webcam frames and writes each
zone's value to the SlowDash datastore as a normal time-series channel.

## What's in here

```
WebcamOCR/
├── SlowdashProject.yaml          # data_source + slowtask + sd_agent.py user module
├── config/
│   ├── slowtask-webcam_ocr.py    # symlink → ../../../slowdash-agent-tools/slowtask/...
│   ├── slowagent-Omega.json      # capture params, prompt, channel list (edit me!)
│   ├── slowplot-Omega.json       # auto-generated by the slowtask from seen channels
│   └── slowdash-Home.json        # home-page layout that includes slowagent in the catalog
├── last_images/                  # populated at runtime; one batch of 14 frames at a time
└── WebcamOCR.db                  # created on first extract; persists across restarts
```

The `slowtask-webcam_ocr.py` symlink points at the submodule — there is no
slowtask copy to keep in sync. The `slowplot-Omega.json` is regenerated by
the slowtask each time a new channel first appears, listing only the
channels actually seen in the data.

## Live setup (Omega CN1500 + Raspberry Pi camera)

Pre-requisite: the Pi at `10.0.0.91` runs a `camera_loop.service` that
captures one JPEG per second to `~sanshiro/public_html/photo.jpg`.

`config/slowagent-Omega.json`'s `capture.source` reads that file directly:

```json
"capture": {
  "source":           "http://10.0.0.91/~sanshiro/photo.jpg",
  "cycle_seconds":     30,
  "frames_per_cycle":  14,
  "frame_interval":    1.5,
  "batch_dir":         "last_images"
}
```

Every 30 seconds the slowtask wipes `last_images/`, captures 14 fresh
frames spaced 1.5 s apart (so each batch spans ~21 s — long enough to
sample every channel as the controller auto-scans through them), downsizes
each frame to 1280-px longest edge, and sends the whole batch to Claude in
one call.

> **Why we don't use `photo.cgi`** — the Pi runs a long-lived
> `rpicam-still --timelapse 1000` that holds the camera continuously. Any
> CGI that tries to acquire it fresh (the original `photo.cgi`) returns a
> Python traceback as the body. Reading the static `photo.jpg` that the
> timelapse already keeps fresh is faster, simpler, and never contends.

## Channel-suffix mapping (OMEGA CN1500 specific)

The CN1500's display auto-scans through enabled channels, showing each
one as `Ctr# Pr` (label, briefly) followed by the temperature and a
**2-character engineering-units suffix**. The user has customised the
suffixes so each channel has a unique tag in the second character:

| Suffix on display | Channel name | Color in plot |
|-------------------|--------------|---------------|
| `C A`             | `ctr1`       | teal          |
| `C b`             | `ctr2`       | orange        |
| `C C`             | `ctr3`       | blue          |
| `C d`             | `ctr4`       | red           |
| `C E`             | `ctr5`       | purple        |
| `C F`             | `ctr6`       | green         |
| `C G`             | `ctr7`       | yellow        |

The first `C` is always Centigrade; the SECOND character is the channel
tag. The `Ctr# Pr` label that flashes just before the value is a
secondary confirmation. The full prompt (in `slowagent-Omega.json`)
walks Claude through this rule explicitly.

## Run

```bash
# 1.  Make sure the headless `claude` CLI is installed and signed in.
#     Vision/OCR is invoked via `claude -p ...` and uses your Claude Code
#     subscription — no Anthropic API key needed.  See
#     https://claude.com/claude-code, then:
claude auth login    # one-time, only if you haven't already

# 2.  Start slowdash from this directory
cd ExampleProjects/WebcamOCR
slowdash --port=18881

# 3.  Open in a browser
#     http://localhost:18881/                                                ← home page (catalog auto-finds slowagent)
#     http://localhost:18881/slowagent.html?config=slowagent-Omega.json      ← agent dashboard
```

## Dashboard

`/slowagent.html?config=slowagent-Omega.json` shows:

- **Recent Frames (left, top)** — auto-cycles through the 14 most-recent
  capture frames, ~1.5 s per frame. Updates as new batches arrive.
- **Extraction Prompt (left, mid)** — editable textarea with two buttons:
  - **Save Prompt** writes the new prompt to `slowagent-Omega.json`. The
    slowtask reloads the config within ~10 s without restarting.
  - **Force Refresh** runs the LLM right now on the frames in
    `last_images`, bypassing the dedup cache. Useful for testing a new
    prompt without waiting for the next 30 s cycle.
- **Example Prompt (left, bottom)** — read-only reference, useful as a
  starting point when authoring a new prompt.
- **Live Plots (right)** — embedded `slowplot.html?config=slowplot-Omega.json`.
  The plot automatically lists only the channels the LLM has actually
  read; new channels are added on-the-fly as they first appear. Default
  range is 24 hours so historical data stays visible during quiet periods.

## Editing on the fly

- Click **Save Prompt** to push prompt edits live (the slowtask polls the
  layout file mtime).
- Click **Force Refresh** to trigger an immediate LLM run on the current
  batch, bypassing dedup. Useful for testing prompt changes.
- Edit `slowagent-Omega.json` directly to change the camera URL, capture
  cadence, model, channels, or per-channel labels/colors. The slowtask
  picks it up on the next ~10 s poll.
- `slowplot-Omega.json` is **auto-managed by the slowtask** — it gets
  rewritten each time a new channel first appears, listing only the
  seen channels. Hand edits will be overwritten on the next channel-set
  change. To keep custom plot styling, edit the channel `color` / `label`
  fields inside `slowagent-Omega.json` instead.

## Data flow

1. Slowtask wakes up every `cycle_seconds`.
2. Wipes `last_images/`, captures `frames_per_cycle` frames from
   `capture.source`, downsizes them, saves with timestamped names.
3. Sends the whole batch to Claude. Response comes back as JSON like
   `{"ctr1": 32.0, "ctr6": 22.0, "ctr2": null, …}`.
4. Each non-null value is appended to `WebcamOCR.db`'s `data` table with
   `channel = "ctr1" | "ctr6" | …` and `timestamp = now`.
5. If a previously-unseen channel just landed, the slowtask:
   - regenerates `slowplot-Omega.json` to add the new line, and
   - hits `/api/channels?force_rescan=true` so slowdash picks up the
     new channel immediately (without it, slowdash's cached schema would
     still report only the old channels until a restart).
6. If the next batch's frames are byte-identical to the previous one
   (e.g. a static camera, frozen instrument), the LLM call is **skipped**
   and the previous values are re-written with the current timestamp so
   the plot keeps a continuous trace.

## Persistence guarantees

- `WebcamOCR.db` is opened in slowpy's append mode (`recreate=False`).
  Rows survive `slowdash` restarts.
- The slowtask never deletes the DB. The only way to wipe history is to
  remove the file manually (`rm WebcamOCR.db`).
- On startup the slowtask logs every channel already in the DB
  (`slowagent: N channel(s) already in DB: …`), so you can verify in the
  log that history loaded.

## Security notes

- LLM auth lives in your Claude Code install (typically OAuth in your
  keychain). This project reads no Anthropic API key — and the subprocess
  is spawned with `ANTHROPIC_API_KEY` stripped from its env so a stale
  key in your shell can't accidentally redirect billing to per-token API.
- The `claude -p` subprocess is launched with `--tools Read` (only the
  Read tool is exposed to Claude) and `--add-dir` pointing at a per-call
  temp directory containing the current batch. Claude has no access to
  the rest of your project, the slowdash DB, or anything else on disk.
- Captured frames live only in process memory, inside `last_images/`,
  and inside the per-call temp directory (auto-deleted when the
  subprocess returns). At the start of every cycle `last_images/` is
  wiped and refilled with the new batch — the previous batch's files
  are deleted before any new frames are written.
- The LLM is asked for JSON values, never for code. The
  `slowagent.sandbox` module is scaffolded for a future advanced mode but
  is not used by this slowtask.

## Troubleshooting

- **`captured 0/14 frame(s)`, log shows "non-image data — skipping"** —
  the camera HTTP endpoint is returning a non-image response (e.g. a
  busy-camera traceback from picamera2). Check that `curl
  http://10.0.0.91/~sanshiro/photo.jpg` returns `image/jpeg` and not
  `text/plain`. If `photo.jpg` itself is broken, SSH to the Pi and check
  `systemctl status camera_loop.service`.
- **Plot is empty even though DB has rows** — slowdash's channel cache
  was empty when it scanned at startup. Trigger a rescan once with
  `curl 'http://localhost:18881/api/channels?force_rescan=true'`. The
  slowtask does this automatically when a new channel first appears, but
  if the rows pre-date the slowtask install, do it once manually.
- **`extracted 0/7 channels`** — Claude couldn't read any value cleanly
  from the batch. Common causes: most frames captured the controller
  during the `Ctr# Pr` label phase rather than the value phase, or the
  display was in a transient state (HELP / a setpoint edit). Just wait
  for the next cycle, or click **Force Refresh** to retry.
